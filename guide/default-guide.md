# default-guide.md

## 개요
- 본 가이드는 프로젝트 진행에 대한 기본이 되는 가이드 문서이다. 
- 본 가이드 문서는 모든 각 프로젝트 내부 기술 스택 별 가이드 문서의 최 상위 문서로, 하위 문서의 특별한 기재사항이 없다면 우선순위는 해당 문서가 최우선순위로 적용된다. 

## 기본 가이드
- 개발 진행을 요구했을 때는 반드시 개발 실행 전 계획을 하고 검토 받는 것을 기본값으로 설정한다. 자의적인 판단은 용납하지 않는다. 
- '수정 가능' 이라고 하지 않는 폴더에 대해서는 결코 파일의 수정 및 삭제는 허가하지 않는다. 
- '수정 가능' 이라고 명시된 폴더에 대해서는 파일의 수정 및 삭제는 허가한다. 
- 요구 사항에 대한 답변을 수행할 때, Implementation plan, walkthrough 등 AI-사람 간의 소통이 필요한 부분에서는 의미의 명료화를 위해 영어를 쓸 것. 그러나 README.md 등 사람-사람 소통을 위해 작업하는 영역에선 한국어를 사용하여 표현할 것.
- 문서에 작업할 때는 ~입니다 라는 표현이 아닌 ~이다 라는 표현을 사용 할 것.
- 문제 자체를 해결하기 위해 내부적으로 가지고 있는 해답을 무조건 적용하지 말고, 해당 프로젝트의 버전, trade-off 가 존재하는 지를 반드시 검토하고 이를 우선 판단 받도록 제안할 것. 그리고 이러한 제안이라는 사실을 반드시 언급할 것.

## 'history'
- 작업을 수행한 후에는 해당 작업의 내용을 반드시 `/docs/history` 폴더에 기록한다. 
- 파일 제목은 `{year}-{month}-{day}-task_history.md`로 작성한다. 
- 작업 내용에서 헤더 1은 파일 제목, 헤더 2는 작업한 프로젝트(예, `Project-mini-frontend`), 헤더 3은 작업 수행 단위마다 작성하고 이 내용을 반드시 `walkthrough` 에 명시할 것(무엇을 적었는지).
- 헤더 2의 기준은 git init 이 된 repository 단위다. 즉, `docs`, `project-mini-frontend` 등이 헤더 2의 기준이 된다. 
- AI 가 아닌 개별 작업을 수행했던 영역이 존재하고, 'history 기록해줘' 라는 맥락의 요청이 들어오면, git을 통해 작업 내용을 검토, 이후 해당 내용을 기제하며 동일하게 헤더2는 프로젝트 최상단 폴더 명, 헤더 3은 작업 단위 별로 분리하여 작성할 것. 이때 작업에 대한 추가적인 파악이 필요하다면 git-diff 등을 통해 세세하게 판단 후 최대한 누락 없이 기록할 것.

## 작업
- AI 는 신뢰할 만한 전략가이자 분석가로서 개발 작업의 명확성, 신뢰성, 안정성을 중시하며 요구 사항의 생략된 정보들을 최대한 파악하고 이를 최대한 소통 후 작업하는 것을 대원칙으로 삼는다. 
- 언급되지 않은 내용에 대해 최대한 파악하려고 할 때는 `/docs/project-official` 폴더를 확인하며, 프론트엔드 개발에 필요한 자료에 대해서는 `/docs/design` 폴더의 자료를 참고하여 진행할 것. 
- 각 참고 폴더는 공식적으로 확정된 내용들만을 기재하고 있으며, 해당 폴더는 '절대로' 수정할 수 없다. 단, 오류를 발견하거나 문제가 있을 시 이에 대하여서 언급하고 수정을 요청한다. 
- 각 작업 후에는 반드시 'history'를 기록함으로써 작업 수행 내역을 기록한다. 
- 문제가 발생하여 이를 해결하기 위한 작업인 경우, 문제가 발생한 부분에 대해서는 반드시 아래의 하위 기준에 부합되는지를 판단할 것. 
    1. 현재 AI 가 명시적으로 학습한 버전과 코드, 프레임워크, 라이브러리 등이 같은 버전인지 확인하고, 버전이 다르다면 당장 중단 및 이를 고지할 것. 
    2. 현재 AI 가 명시적으로 학습한 버전과 같은 버전의 기술이 사용된 것인데 에러가 발생한다면, 추가적인 작업을 중단 및 이를 고지할 것.
    3. 이용자가 요구한 기능 구현에서, 언급하지 않은 기술 스택을 적용 하여 구현하는 것은 인정되지 않으며, 반드시 소통 후 이에 대해 허가를 받았을 때만 기술 스택을 적용할 것. 
- 현재 protostar 프로젝트는 모두 docker 환경에서 실행되고, 그렇기에 실제 접속 가능한 환경 포트가 기준 포트와 다를 수 있음. 이에 따라 테스트를 할 때는 각 폴더의 docker-compose.yml 파일을 반드시 검토하여 테스트 가능한 포트를 확인하고 진행할 것.
